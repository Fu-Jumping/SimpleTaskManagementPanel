# 测试与报告任务单

本文档给测试与报告负责人使用。测试和报告必须以实际运行结果为准，截图来自真实项目页面，报告只描述最终完成且可演示的功能。

## 一、职责范围

测试与报告负责人负责：

- 前后端联调走查
- 演示流程测试
- 问题记录
- 页面截图
- 实验报告整理
- 最终提交材料检查

开发或文档分支：

```text
feature/test-report
```

说明：

- 测试与报告负责人最后集中介入，但建议提前阅读 `README.md`、`docs/requirements.md` 和 `docs/api.md`。
- 发现问题后记录清楚复现步骤，再交给对应开发负责人修复。

## 二、输入材料

测试前需要确认已拿到：

- 项目代码
- 前端启动说明
- 后端启动说明
- 接口契约：`docs/api.md`
- 需求与验收说明：`docs/requirements.md`
- 分工说明：`docs/分工/分工说明.md`
- 原型参考说明：`docs/design-reference.md`

如果项目无法启动，需要先记录启动失败原因，例如：

- 依赖未安装
- 环境变量缺失
- 端口被占用
- 前端接口地址配置错误
- 后端数据库文件缺失

## 三、测试环境

建议环境：

- Node.js LTS 版本
- npm
- Chrome 或 Edge 浏览器

端口约定：

- 前端：`http://localhost:5173`
- 后端：`http://localhost:3000`

启动后先确认：

- 前端页面可以打开。
- 后端接口服务可以启动。
- 浏览器控制台无持续报错。
- 后端终端无持续报错。

## 四、联调测试流程

### 1. 注册测试

步骤：

1. 打开注册页。
2. 输入新用户名和密码。
3. 点击注册。

预期：

- 注册成功。
- 页面跳转登录页或主看板页。
- 如果直接进入主看板，应显示当前用户信息。

记录：

- 注册页截图。
- 注册成功后的页面截图。

### 2. 登录测试

步骤：

1. 打开登录页。
2. 输入已注册用户名和密码。
3. 点击登录。

预期：

- 登录成功。
- 进入任务看板。
- 用户菜单显示用户名。

补充测试：

- 输入错误密码，应该登录失败并显示提示。

### 3. 任务 CRUD 测试

新增：

1. 点击新建任务。
2. 填写标题、描述、截止日期、优先级、状态。
3. 保存。

预期：

- 任务出现在对应状态列。

编辑：

1. 打开任务编辑抽屉。
2. 修改标题、描述、优先级或状态。
3. 保存。

预期：

- 页面显示修改后的内容。

删除：

1. 点击删除按钮。
2. 确认删除。

预期：

- 删除前出现确认框。
- 确认后任务消失。

快捷键删除：

1. 点击选中任务。
2. 按 Del。
3. 确认删除。

预期：

- 弹出同样的删除确认。
- 确认后任务删除。

### 4. 状态看板拖拽测试

步骤：

1. 在状态看板中将任务从待处理拖到进行中。
2. 将任务从进行中拖到已完成。
3. 在同一列内调整顺序。
4. 刷新页面。

预期：

- 状态变化正确。
- 排序变化正确。
- 刷新后结果保留。

### 5. 优先级看板拖拽测试

步骤：

1. 切换到优先级看板。
2. 将任务从中优先级拖到高优先级。
3. 查看任务状态是否保持不变。
4. 刷新页面。

预期：

- 优先级变化正确。
- 状态不被优先级拖拽改变。
- 刷新后结果保留。

### 6. 搜索筛选测试

步骤：

1. 输入关键词。
2. 选择状态筛选。
3. 选择优先级筛选。
4. 组合多个筛选条件。
5. 清空筛选。

预期：

- 搜索结果数量正确。
- 标题和描述中的关键词高亮。
- 组合筛选结果正确。
- 清空筛选后恢复全部任务。

### 7. 导入导出测试

导出：

1. 设置筛选条件。
2. 点击导出。
3. 查看导出的 JSON。

预期：

- 导出格式包含 `exportedAt`、`count`、`tasks`。
- `count` 与当前筛选结果一致。

导入：

1. 准备包含合法任务和非法任务的 JSON 文件。
2. 点击导入。
3. 查看导入反馈。

预期：

- 合法任务被导入。
- 非法任务被跳过。
- 页面显示成功数量和跳过数量。

### 8. 登录状态和持久化测试

步骤：

1. 登录后新增任务。
2. 刷新页面。
3. 关闭并重新打开页面。
4. 退出登录。
5. 尝试访问主看板页。

预期：

- 刷新后任务仍存在。
- 登录状态按设计恢复或重新登录。
- 退出登录后不能直接访问主页面。

## 五、问题记录格式

建议记录到 `docs/test-record.md` 或报告草稿中。

格式：

```text
问题编号：
发现时间：
发现人：
页面或接口：
复现步骤：
实际结果：
预期结果：
严重程度：
负责人：
修复状态：
备注：
```

严重程度建议：

| 等级 | 说明 |
| --- | --- |
| 高 | 阻塞主流程演示 |
| 中 | 功能可用但明显影响体验 |
| 低 | 文案、样式或轻微边界问题 |

## 六、截图清单

报告建议截图：

- 登录页
- 注册页
- 状态看板
- 优先级看板
- 搜索结果
- 新建任务抽屉
- 编辑任务抽屉
- 删除确认弹窗
- 数据中心

截图要求：

- 页面内容尽量使用中文任务数据。
- 截图中不要出现浏览器控制台报错。
- 截图要能支撑报告中的功能说明。

## 七、报告结构

建议报告标题：

```text
简易任务管理面板实验报告
```

建议结构：

1. 题目要求
2. 系统概述
3. 技术选型
4. 系统设计
5. 功能实现
6. 接口说明
7. 测试验证
8. 小组分工
9. 总结

篇幅建议：

- 2500-4000 字。
- 不需要过度展开实现细节。
- 重点说明系统功能、接口设计、测试结果和分工。

## 八、报告内容要点

题目要求：

- 简述 Vue 3 + Vite、拖拽、任务 CRUD、导入导出、后端 API 等要求。

系统概述：

- 说明项目是个人任务看板系统。
- 说明用户可以通过看板管理任务状态和优先级。

技术选型：

- 前端 Vue 3、Vite、Pinia、Ant Design Vue、vuedraggable。
- 后端 Node、Express、lowdb、JWT、bcrypt。

系统设计：

- 前后端分离。
- 前端负责页面、路由、状态管理和交互。
- 后端负责认证、任务接口、数据持久化和用户隔离。

功能实现：

- 登录注册。
- 任务 CRUD。
- 状态看板。
- 优先级看板。
- 搜索筛选。
- 导入导出。
- 数据持久化。

接口说明：

- 可引用 `docs/api.md` 的主要接口。
- 不需要粘贴全部接口 JSON，挑关键接口说明即可。

测试验证：

- 使用表格列出测试项、操作步骤、预期结果、实际结果。
- 配合截图说明。

小组分工：

- 按个人任务记录贡献。
- 成员姓名确定后再替换角色占位。

总结：

- 总结完成情况、遇到的问题和改进方向。

## 九、最终提交检查

提交前确认：

- 代码包含前端和后端。
- README 有启动说明。
- 接口文档存在。
- 报告存在。
- 报告截图完整。
- 项目能在本地启动。
- 演示清单主流程可通过。
- 报告内容与实际功能一致。

## 十、降级记录

如果有功能被隐藏或未完成，需要记录：

```text
功能名称：
原计划：
实际处理：
降级原因：
是否影响主流程：
报告是否提及：
```

原则：

- 主流程功能尽量不降级。
- 非核心功能不稳定时可以隐藏入口。
- 报告只写实际可演示的内容。
