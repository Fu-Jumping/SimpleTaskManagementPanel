# 作业2小组分工说明

## 一、项目目标

本项目从零开始完成一个个人任务看板系统，重点保证演示链路完整、界面效果清晰、功能稳定可用。

核心功能包括：

- 用户注册、登录、退出登录
- 任务新增、查看、编辑、删除
- 状态看板：待处理、进行中、已完成
- 优先级看板：高、中、低
- 拖拽修改任务状态、优先级和排序
- 关键词、状态、优先级组合搜索筛选
- 数据导入、导出
- 登录后数据持久化，刷新页面后数据仍保留

开发原则：

- 优先保证主流程稳定，影响演示稳定的加分功能可以降级或隐藏入口。
- 前后端以接口契约为准，字段命名统一使用英文驼峰。
- 项目代码前后端同仓管理，目录暂定为 `client/` 和 `server/`。

## 二、总体分工

小组采用“规划与原型 / 前端 / 后端 / 测试报告”的分工方式。


| 角色       | 人数  | 主要职责                                             |
| -------- | --- | ------------------------------------------------ |
| 规划与原型负责人 | 1 人 | 创建主仓库和项目空骨架；整理需求规划、原型说明、接口契约占位；开发阶段负责需求解释和最终验收把关 |
| 前端负责人    | 1 人 | 完成 Vue 前端项目，实现页面、路由、状态管理、拖拽、快捷键、API 接入和前端错误提示    |
| 后端负责人 A  | 1 人 | 主责 Express 服务、认证接口、JWT、任务接口、导入导出接口和接口说明维护        |
| 后端负责人 B  | 1 人 | 主责 lowdb 数据结构、用户任务隔离、示例数据、接口验证，并协助后端 A 修改问题      |
| 测试与报告负责人 | 1 人 | 负责前后端联调走查、截图、问题记录、实验报告整理和最终提交材料检查                |


说明：

- 具体成员姓名后续确定后，可补充到上表。
- 后端 A/B 是责任划分不是严格文件边界，两名后端同学可以共同修改后端代码。
- 测试与报告负责人最后介入联调，但开发阶段可提前查看需求、接口和运行说明，避免后期无法启动项目。

## 三、各角色详细职责

### 1. 规划与原型负责人

主要职责：

- 初始化 GitHub 主仓库和项目空骨架。
- 建立基础目录，例如 `client/`、`server/`、`docs/`。
- 整理需求规划、功能范围、数据字段和验收清单。
- 准备视觉原型或原型说明，作为前端实现参考；原型素材保留在根目录 `design/`。
- 创建接口契约占位文件，例如 `docs/api.md`。
- 开发过程中统一解释需求，判断是否需要调整或裁剪功能。
- 最终按演示清单进行验收，确认主链路是否稳定。

最低交付物：

- 项目仓库空骨架
- README 初稿
- 分工说明文档
- 需求规划文档
- 接口契约占位
- 原型说明或原型图，参考 `docs/design-reference.md` 和根目录 `design/`

### 2. 前端负责人

技术范围：

- Vue 3
- Vite
- Vue Router
- Pinia
- Ant Design Vue
- vuedraggable

主要职责：

- 初始化并完成 `client/` 前端项目。
- 实现登录页、注册页、主看板页、数据中心入口等页面。
- 实现状态看板和优先级看板。
- 实现任务新增、编辑、删除抽屉。
- 实现任务拖拽排序、状态变更和优先级变更。
- 实现关键词、状态、优先级组合筛选。
- 实现标题和描述关键词高亮。
- 实现任务选中后按 Del 删除，并统一弹出删除确认。
- 封装统一 API 请求模块，处理 token 注入、错误提示和响应解包。
- 前期可按接口契约使用 Mock 数据开发，后期切换真实后端接口。
- 实现页面前参考根目录 `design/` 和 `docs/design-reference.md`，但不直接使用导出代码作为正式代码起点。
- 保证移动端基础可用，无明显布局溢出。

最低交付物：

- 可运行的 Vue 前端项目
- 主要页面和交互实现
- Mock 数据或真实接口接入代码
- 前端启动说明

### 3. 后端负责人 A：接口与认证

技术范围：

- Node.js
- Express
- JWT
- bcrypt
- lowdb

主要职责：

- 初始化并维护 `server/` 后端项目结构。
- 实现 Express 服务启动和基础中间件。
- 实现注册、登录接口。
- 使用 bcrypt 存储密码哈希。
- 使用 JWT 完成基础登录认证，有效期为 7 天。
- 实现任务 CRUD 接口。
- 实现任务批量排序接口，支持状态看板和优先级看板拖拽同步。
- 实现导入、导出接口。
- 维护 `docs/api.md` 接口说明，包括路径、方法、请求字段、响应示例和说明。

最低交付物：

- 可运行的 Express 后端项目
- 认证接口
- 任务基础接口
- 排序、导入、导出接口
- 接口说明文档

### 4. 后端负责人 B：数据与验证

主要职责：

- 设计和维护 lowdb 数据结构。
- 确保 `db.json` 中用户和任务数据结构清晰。
- 实现或协助实现用户任务隔离。
- 实现或协助实现新用户首次登录生成示例任务。
- 实现或协助实现用户菜单追加示例数据的后端支持。
- 按接口说明测试后端接口。
- 重点验证注册、登录、鉴权失败、任务 CRUD、拖拽排序、导入导出。
- 发现接口问题后协助后端 A 修改。

最低交付物：

- lowdb 数据结构实现
- 用户数据隔离逻辑
- 示例任务逻辑
- 后端接口测试或验证记录

### 5. 测试与报告负责人

主要职责：

- 根据演示清单进行前后端联调走查。
- 记录运行过程中的问题、复现步骤和修复状态。
- 截取报告所需页面截图。
- 整理实验报告初稿。
- 检查最终提交材料是否包含代码和报告。
- 报告内容必须与实际项目一致，截图和功能说明不能脱离真实实现。

测试重点：

- 注册
- 登录
- 任务新增
- 任务编辑
- 任务删除
- 状态看板拖拽
- 优先级看板拖拽
- 搜索筛选
- 导入
- 导出
- 刷新页面后数据保留
- 退出登录

报告建议结构：

- 题目要求
- 系统设计
- 功能实现
- 接口说明
- 测试验证
- 分工总结

## 四、GitHub 协作规则

主仓库放在规划与原型负责人的 GitHub 账号下，其他成员加入 collaborator，拥有写权限。

分支规则：


| 分支                    | 使用人       | 用途              |
| --------------------- | --------- | --------------- |
| `main`                | 全组        | 主分支，保存阶段性可运行版本  |
| `feature/client`      | 前端负责人     | 前端功能开发          |
| `feature/server`      | 后端负责人 A/B | 后端功能开发          |
| `feature/test-report` | 测试与报告负责人  | 测试记录、报告、截图等材料整理 |


协作要求：

- 所有成员从 `main` 拉取自己的功能分支。
- 开发完成一个相对完整的模块后再提交代码。
- 阶段完成后可自行合并回 `main`。
- 合并前至少确认项目能启动，不能有明显运行报错。
- 合并前可使用 AI 或其他工具自查代码。
- 如果修改了接口路径、字段或响应结构，必须同步更新 `docs/api.md`。
- 如发生合并冲突，原则上由对应开发者自行处理，必要时找规划与原型负责人统一判断。

## 五、接口与数据约定

接口响应格式统一：

成功：

```json
{
  "success": true,
  "data": {}
}
```

失败：

```json
{
  "success": false,
  "message": "错误信息"
}
```

任务字段统一使用英文驼峰：


| 字段            | 说明                           |
| ------------- | ---------------------------- |
| `id`          | 任务 ID                        |
| `title`       | 任务标题                         |
| `description` | 任务描述                         |
| `dueDate`     | 截止日期                         |
| `priority`    | 优先级，可选 `high`、`medium`、`low` |
| `status`      | 状态，可选 `todo`、`doing`、`done`  |
| `order`       | 列内排序                         |
| `createdAt`   | 创建时间                         |
| `updatedAt`   | 更新时间                         |


主要接口暂定：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/tasks`
- `POST /api/tasks`
- `PATCH /api/tasks/:id`
- `DELETE /api/tasks/:id`
- `PATCH /api/tasks/reorder`
- `POST /api/tasks/import`
- `GET /api/tasks/export`

最终接口以 `docs/api.md` 为准。

## 六、阶段安排

### 第 1 阶段：骨架与契约

目标：

- 创建主仓库和基础目录。
- 写好 README、分工说明、需求规划和接口契约占位。
- 前后端明确字段、接口和页面范围。

负责人：

- 规划与原型负责人主责
- 前端、后端阅读确认

### 第 2 阶段：前后端并行开发

目标：

- 前端按原型和 Mock 数据开发页面。
- 后端按接口契约实现认证、任务、排序、导入导出。
- 后端维护接口文档，前端按接口文档对接。

负责人：

- 前端负责人
- 后端负责人 A/B

### 第 3 阶段：真实接口联调

目标：

- 前端从 Mock 切换到真实接口。
- 修复字段不一致、鉴权、拖拽同步、导入导出等问题。
- 保证主链路完整可演示。

负责人：

- 前端负责人主责 API 接入
- 后端负责人 A/B 配合修复接口问题
- 测试与报告负责人开始记录问题

### 第 4 阶段：验收与报告

目标：

- 按演示清单完整走查。
- 截取报告截图。
- 整理实验报告。
- 最终确认提交材料。

负责人：

- 测试与报告负责人主责
- 规划与原型负责人验收把关
- 前端、后端修复必要问题

## 七、验收清单

最终演示至少需要通过以下流程：

- 可以注册新用户。
- 可以登录已有用户。
- 未登录访问主页面会跳转登录页。
- 已登录访问登录或注册页会跳转看板页。
- 可以新增任务。
- 可以编辑任务。
- 可以删除任务，按钮删除和 Del 快捷键删除都需要确认。
- 状态看板可以拖拽任务改变状态和排序。
- 优先级看板可以拖拽任务改变优先级和排序。
- 优先级拖拽只改变 `priority`，不改变 `status`。
- 可以按关键词、状态、优先级组合筛选任务。
- 搜索结果显示数量，并高亮标题和描述中的关键词。
- 可以导入任务，非法项跳过，并显示成功数量和跳过数量。
- 可以导出当前筛选任务。
- 刷新页面后数据仍保留。
- 可以退出登录。
